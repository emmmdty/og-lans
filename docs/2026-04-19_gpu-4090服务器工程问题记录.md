## 2. 本次暴露的工程问题

### 2.1 远端主仓不是干净副本

排查时，`~/masterProjects/og-lans` 停留在旧提交 `c82345b`，且主工作区带未提交改动，不能直接盲目覆盖或直接开始训练。

### 2.2 远端 `.worktrees` 体积过大且存在陈旧元数据

- 远端 `.worktrees` 目录占用约 `5.5G`
- 远端 `git worktree list --porcelain` 同时包含：
  - 仓库内旧 worktree
  - 指向本地 Linux 路径 `/home/tjk/...` 的失效 worktree 记录

这说明远端 repo 不能继续被当作“可直接 rsync 覆盖的轻量副本”使用，必须先归档再收敛。

### 2.3 不能假设服务器上有 `uv`

本仓库本地默认工作流是 `uv sync` / `uv run`，但远端没有 `uv`。如果直接复用本地习惯命令，服务器会在入口阶段失败。

结论：服务器执行必须显式使用

```bash
/home/TJK/.conda/envs/tjk/bin/python
```

而不是依赖 `python`、`python3`、`uv` 或 `conda activate`。

### 2.4 旧版同步配置没有排除 `.worktrees`

`sync-remote.yaml` 在本次修复前没有把 `.worktrees` 排除在同步 payload 外。这会把陈旧 worktree 一并推送到服务器，导致：

- 同步体积不必要地放大
- 远端 repo 状态进一步污染
- 后续长时间训练和评测的工作目录更难稳定

本次已在本地 `sync-remote.yaml` 中修复：`sync.sync.excludes` 现在明确包含 `.worktrees`。

说明：

- `sync-remote.yaml` 当前是本地忽略文件，不随 Git 提交；
- 因此它只约束本机后续的 `sr upload`/rsync 工作流，不会自动同步到服务器。

### 2.5 学术评测脚本错误提示了不存在的安装方式

`scripts/run_eval_academic.sh` 之前在缺依赖时提示：

```bash
pip install -r requirements.txt && pip install -e .
```

但仓库并没有 `requirements.txt`。这会直接误导服务器排障。

本次已修复为：

- `uv sync --extra dev`
- 或 `pip install -e .`

## 3. 后续推荐操作口径

### 3.1 代码同步

对 `gpu-4090`，优先使用以下流程：

1. 本地代码修复并通过测试
2. 本地生成 `git bundle`
3. 远端先归档当前 repo 状态到 `logs/remote_sync_archive/<timestamp>/`
4. 远端 `git fetch <bundle>` + `git reset --hard`
5. 远端清理 `.worktrees`、缓存和陈旧同步产物

不推荐直接使用 `sr upload` 作为本轮主同步方式，除非已经确认：

- 本地同步配置排除了 `.worktrees`
- 远端 repo 已经是干净副本
- 不需要同步 `.git` 历史和提交对象
