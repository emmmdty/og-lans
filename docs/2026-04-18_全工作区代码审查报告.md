# OG-LANS 全工作区代码审查报告（2026-04-18）

## 审查范围与方法

本轮审查覆盖 9 个 checkout：

- `main` 当前工作区状态，包含未提交改动。
- `.worktrees/candidate-table-v2`
- `.worktrees/candidate-table-v2-recall`
- `.worktrees/event-probe-v1`
- `.worktrees/event-probe-v2-error-driven`
- `.worktrees/record-assembly-v1`
- `.worktrees/server-storage-migration`
- `.worktrees/sft-distill-role-span-table`
- `.worktrees/structured-sft-pipeline`

方法约束：

- 全部审查均为只读，不修改代码，不回退任何现有改动。
- 每个 checkout 均使用一个独立 subagent 审查，模型统一为 `gpt-5.4`，推理强度统一为 `xhigh`。
- 结论以工程性问题为主，优先记录 bug、回归风险、契约漂移、脚本入口风险、provenance/manifest 不可靠、以及测试缺口。
- 除 `main` 的部分共享链路做过定向 `pytest` 核验外，其余 worktree 以静态审查为主。

## 总体结论

这 9 个 checkout 都存在代码工程性问题，但问题类型并不相同。

最值得优先处理的不是某一个孤立分支，而是几条跨工作区复现的共享根因：

1. 训练、评测、审计三条共享链路没有形成稳定闭环。
2. 训练入口、wrapper、run manifest、resume 语义之间存在多处名实不符。
3. rerank sidecar 和结构化 SFT 路径没有被正式评测/复现入口完整吸收，导致“结果可跑，但不可审计”。
4. 多个 worktree 在 helper、canonicalization、repair、assembly、cache/runtime 约束上各自再实现一遍，已经出现明显漂移。

如果只按“哪条实验线看起来最 promising”来推进，而不先补这些共享工程口子，后续结果比较会持续掺入协议差异、运行时差异和产物记录差异。

## 共享工程风险

### 1. `two_stage_typed` 训练/评测/审计契约没有闭环

- `main` 中训练侧把 `two_stage_typed` 按 `two_stage` 展开，仍只生成单条 stage2 样本，也没有携带 `target_event_types`，见 `src/oglans/utils/training_protocol.py:183`。
- 评测侧却会按事件类型逐条执行 stage2，并在 API 调用时显式传入 `target_event_types`，见 `evaluate_api.py:756`、`evaluate_api.py:779`。
- 审计链路仍以旧假设统计 few-shot/typed 结果，见 `scripts/audit_baseline_matrix.py:97`。
- 结果是：训练学的不是实际推理任务，summary/audit 又不能准确反映 typed 路线。

### 2. compare contract 没有把 semantic version 漂移纳入 fail-fast

- `main` 已经在 `run_manifest` / summary 里写入 semantic version 元数据，见 `src/oglans/utils/run_manifest.py:202`、`evaluate_api.py:1711`。
- 但 compare contract 仍不包含 prompt/parser/normalization 版本，见 `src/oglans/utils/compare_contract.py:13`。
- API multi-seed 汇总只按 compare hash 合并，见 `scripts/run_api_repro_suite.py:493`。
- 结果是：不同 worktree 或未提交版本漂移仍可能被当作“同协议结果”直接汇总。

### 3. `train_fit` 被重复应用，`teacher_silver` 默认可能被静默过滤掉

- `sft-distill-role-span-table` 明确暴露出主链路问题：`main.py` 先对 gold 做一次 `select_training_fit_samples()`，再把 `teacher_silver` 拼入训练集；trainer 初始化时又再过滤一遍，见 `.worktrees/sft-distill-role-span-table/main.py:291`、`.worktrees/sft-distill-role-span-table/src/oglans/trainer/unsloth_trainer.py:1480`。
- `teacher_silver` 的样本 id 固定为 `teacher::<source_id>`，默认 frozen split 下不会命中 `fit_ids`，见 `.worktrees/sft-distill-role-span-table/src/oglans/utils/teacher_silver.py:341`。
- 这个根因不是该分支独有，同一链路也存在于 `main` 与 `candidate-table-v2`。

### 4. `resume_from_checkpoint` 的命名和真实行为不一致

- 在 `sft-distill-role-span-table` 和 `structured-sft-pipeline` 中，`resume_from_checkpoint` 实际只是 LoRA 权重热启动，不会恢复 optimizer、scheduler、global step，也没有校验阶段兼容性，见 `.worktrees/sft-distill-role-span-table/src/oglans/trainer/unsloth_trainer.py:1596`、`.worktrees/structured-sft-pipeline/src/oglans/trainer/unsloth_trainer.py:1591`。
- 但 manifest 和脚本又把它记录成“resume”。
- 这会把“恢复训练”和“基于旧权重重开新训练”混在一起，直接影响训练审计和阶段串联。

### 5. `torchrun` / DDP 下的缓存、debug 产物和 provenance 不安全

- `sft-distill-role-span-table` 明确暴露：wrapper 支持 `torchrun`，但 trainer 会在所有 rank 上删除并重建同一个 `dpo_dataset_cache`，也会往同一个 `debug_data_dir` 写文件，见 `.worktrees/sft-distill-role-span-table/scripts/run_train.sh:421`、`.worktrees/sft-distill-role-span-table/src/oglans/trainer/unsloth_trainer.py:1985`、`:2069`。
- 同时 wrapper manifest 在真正启动分布式运行前就落盘，`world_size`、config hash、`sft_task` 等字段都可能和真实训练态不一致，见 `.worktrees/sft-distill-role-span-table/src/oglans/utils/run_manifest.py:223`、`.worktrees/sft-distill-role-span-table/scripts/run_train.sh:281`。
- 同类问题也在 `main` 和 `candidate-table-v2` 的共享链路里出现。

### 6. 评测指标、canonicalization 和诊断产物口径不一致

- `candidate-table-v2` 和 `event-probe-v1` 的 rerank summary 都存在“主指标基于原始 reranked events，诊断基于 canonicalized prediction”的分叉，见 `.worktrees/candidate-table-v2/scripts/rerank_eval_predictions.py:275`、`:344` 与 `.worktrees/event-probe-v1/scripts/rerank_eval_predictions.py:307`、`.worktrees/event-probe-v1/src/oglans/utils/extraction_error_audit.py:208`。
- 一旦 role alias map 真正改写字段，summary 内部就会出现自相矛盾。

### 7. 评测本体本身还有顺序敏感与统计失真

- `event-probe-v1` 指出 `doc_role_micro_f1` 的 record matching 对输入顺序敏感，见 `.worktrees/event-probe-v1/src/oglans/evaluation.py:473`、`:482`。这属于共享 evaluator 风险，`main` 也继承。
- `main` 中 `meta.api_response_models` 汇总路径与真实 `response_meta` 结构不一致，导致 summary 字段长期空转，见 `evaluate_api.py:1450`、`:1638`、`:1727`。

## 各工作区审查结果

### 1. `main`

结论：`main` 的主要问题不在单点实现，而在共享契约没有闭环，已经影响 API 基线结果的可比性和审计可靠性。

- `[高][共享]` `two_stage_typed` 的训练协议、API 评测协议和审计逻辑不一致，见 `src/oglans/utils/training_protocol.py:183`、`evaluate_api.py:756`、`evaluate_api.py:779`、`scripts/audit_baseline_matrix.py:97`。
- `[高][共享]` semantic version 已写入 summary/manifest，但 compare gate 仍不把 prompt/parser/normalization version 纳入 fail-fast，见 `src/oglans/utils/run_manifest.py:202`、`evaluate_api.py:1711`、`src/oglans/utils/compare_contract.py:13`、`scripts/run_api_repro_suite.py:493`。
- `[中][共享]` `meta.api_response_models` 读取路径错误，summary 中该字段几乎恒为空，见 `evaluate_api.py:1450`、`:1638`、`:1727`、`scripts/validate_academic_artifacts.py:46`。
- `[中][共享]` `two_stage_typed` 的 few-shot 审计统计会被系统性低估，见 `evaluate_api.py:559`、`:837`、`src/oglans/utils/compare_contract.py:139`。
- `[低][共享]` `scripts/run_eval_api.sh` 仍用脆弱的字符串拼接加 `bash -lc "$cmd"`，wrapper test 也只做静态断言，见 `scripts/run_eval_api.sh:166`、`:243`、`:302`、`tests/test_script_wrappers.py:146`。

补充：本轮对 `main` 跑过定向测试，相关子集 `66 passed`。这说明问题更多是契约和审计层缺口，而不是当前会直接打红现有测试的显性 bug。

### 2. `candidate-table-v2`

结论：这条线的主要风险不是候选表思想本身，而是训练目标、rerank 接入和 provenance 还没有形成正式闭环。

- `[高][特有]` `training.sft_task=candidate_table` 目前并没有训练“候选表”目标，仍复用了普通 extraction prompt 和 `sample.chosen` 事件 JSON，见 `.worktrees/candidate-table-v2/src/oglans/utils/training_protocol.py:448`、`:550`、`.worktrees/candidate-table-v2/src/oglans/data/adapter.py:23`、`:117`。
- `[高][特有]` rerank 结果没有接入正式评测/复现入口，也不产出等价的 `run_manifest` 与 compare contract，见 `.worktrees/candidate-table-v2/scripts/rerank_eval_predictions.py:328`、`.worktrees/candidate-table-v2/scripts/run_eval_base.sh:189`、`.worktrees/candidate-table-v2/scripts/run_local_repro_suite.py:474`。
- `[中][共享链路]` summary metrics 与 diagnostics 使用不同预测口径，见 `.worktrees/candidate-table-v2/scripts/rerank_eval_predictions.py:275`、`:344`、`.worktrees/candidate-table-v2/src/oglans/utils/extraction_error_audit.py:197`。
- `[中][特有]` `assemble_records_from_span_inventory()` 仍是按索引硬配对的脆弱启发式，而且当前 rerank 流水线并未真正调用它，见 `.worktrees/candidate-table-v2/src/oglans/utils/candidate_assembly.py:164`、`.worktrees/candidate-table-v2/scripts/rerank_eval_predictions.py:22`。
- `[中][特有]` 多个评测脚本仍把 `MODELSCOPE_CACHE` 强行写死为仓库内 `./models`，与存储迁移方向冲突，见 `.worktrees/candidate-table-v2/scripts/run_eval_base.sh:13`、`run_eval_api.sh:14`、`run_eval_academic.sh:19`。
- `[低][特有]` schema/grounding/repair 规则已在 rerank、candidate assembly、teacher_silver 三处重复实现，后续极易漂移，见 `.worktrees/candidate-table-v2/scripts/rerank_eval_predictions.py:55`、`.worktrees/candidate-table-v2/src/oglans/utils/candidate_assembly.py:221`、`.worktrees/candidate-table-v2/src/oglans/utils/teacher_silver.py:146`。

### 3. `candidate-table-v2-recall`

结论：这条 recall 增强线不是“轻微退化”，而是把文档级 regex 候选直接补回记录后，破坏了 v2 原本较保守的组装边界。

- `[高][特有]` recall 在预去重之后执行，可以重新制造重复记录，而且不会再做第二轮去重，见 `.worktrees/candidate-table-v2-recall/scripts/rerank_eval_predictions.py:199`、`:206`、`:290`、`.worktrees/candidate-table-v2-recall/src/oglans/utils/candidate_assembly.py:358`、`:402`、`:406`。
- `[高][特有]` `质押/事件时间` recall 直接取文档中第一个日期，极易与披露时间混淆，见 `.worktrees/candidate-table-v2-recall/src/oglans/utils/candidate_assembly.py:61`、`:65`、`:402`、`:403`。
- `[中][特有]` recall 候选是整篇文档共享的，没有 record-local 归属或“值已被消费”的概念，同一值会被补到多条记录上，见 `.worktrees/candidate-table-v2-recall/src/oglans/utils/candidate_assembly.py:389` 到 `:408`。
- `[中][特有]` `公司上市/环节` 的 recall 选值没有稳定策略，set 候选加“第一个命中”在工程上很脆，见 `.worktrees/candidate-table-v2-recall/src/oglans/utils/candidate_assembly.py:32`、`:153`、`:403`。
- `[低][特有]` 新增 recall 路径的测试只覆盖单记录 happy path，没有覆盖多记录污染、时间角色误补和 CLI 入口，见 `.worktrees/candidate-table-v2-recall/tests/test_candidate_assembly.py:199`、`.worktrees/candidate-table-v2-recall/tests/test_rerank_eval_predictions.py:143`、`:245`。

### 4. `event-probe-v1`

结论：方向是保守的，但当前实现还没有达到“稳定且可审计”的工程标准。

- `[高][共享]` `doc_role_micro_f1` 的文档内 record matching 对输入顺序敏感，见 `.worktrees/event-probe-v1/src/oglans/evaluation.py:473`、`:482`，现有测试也没有覆盖 permutation invariance，见 `.worktrees/event-probe-v1/tests/test_metric_semantics.py:120`。
- `[高][特有]` rerank summary 的主指标与诊断可能针对两套不同预测，见 `.worktrees/event-probe-v1/scripts/rerank_eval_predictions.py:305`、`:307`、`:324`、`.worktrees/event-probe-v1/src/oglans/utils/extraction_error_audit.py:208`。
- `[中][特有]` `extraction_error_audit` 用无序 `(event_type, role, value)` 统计，无法暴露跨 record 组装错误，见 `.worktrees/event-probe-v1/src/oglans/utils/extraction_error_audit.py:34`、`:117`、`:137`。
- `[中][特有]` `解除质押 -> 质押` 的核心 probe 规则依赖 record 列表顺序，而不是具体 record 证据，见 `.worktrees/event-probe-v1/src/oglans/utils/event_probe.py:275`、`:276`。
- `[中][特有]` rerank sidecar 不保存 protocol hash、role alias hash 或 compare-contract hash，见 `.worktrees/event-probe-v1/scripts/rerank_eval_predictions.py:375`，但基础评测路径其实已有这些字段，见 `.worktrees/event-probe-v1/evaluate.py:1213`、`:1224`、`:1233`。

### 5. `event-probe-v2-error-driven`

结论：`v2` 的增益存在，但 `v3` 只是在最外层脚本被挡住，并没有真正隔离。

- `[中][共享]` summary provenance 不足，只记录 `repair_mode` / `event_probe_mode` / `total_samples`，无法区分 worktree-specific `v2` 结果，见 `.worktrees/event-probe-v2-error-driven/scripts/rerank_eval_predictions.py:407`。
- `[低][特有]` 测试当前在保护 quarantined 逻辑本身，而不是保护边界，见 `.worktrees/event-probe-v2-error-driven/tests/test_event_probe.py:241` 到 `:328`、`.worktrees/event-probe-v2-error-driven/tests/test_rerank_eval_predictions.py:430`。

### 6. `record-assembly-v1`

### 7. `server-storage-migration`

结论：这个 worktree 虽然相对 `main` 没有 ahead 提交，但并不是“无独立代码面”；它以落后 `main` 的形式重新暴露出几处工程回退。

### 9. `structured-sft-pipeline`

## 优先级最高的问题清单

按跨工作区影响范围和工程后果排序，当前最值得优先处理的是：

1. `main` 的 `two_stage_typed` 训练/评测/审计契约不一致。
2. compare contract 未纳入 semantic version 漂移，导致跨 worktree 结果可被误汇总。
3. `train_fit` 双重过滤导致 `teacher_silver` 默认可能进不了训练。
4. `resume_from_checkpoint` 名实不符，把 warm-start 冒充 resume。
5. `torchrun` / DDP 下缓存、debug 产物与 wrapper manifest 不安全。
6. `candidate-table-v2` 的 `candidate_table` 训练目标并未真正独立成新任务。
7. `candidate-table-v2-recall` 的文档级 recall 直接破坏 record 级边界。

9. `event-probe-v2-error-driven` 的 `v3` 没有被真正隔离，只是最外层脚本做了软封禁。
10. `structured-sft-pipeline` 的结构化阶段 checkpoint 可以被“正常评测”，但评的是错误任务。
