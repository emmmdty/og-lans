# OG-LANS 全仓问题驱动修复方案（2026-04-19）

## 0. 方案定位

本方案用于指导后续对当前项目代码问题的集中修复。它不是实验结果复盘，也不是逐个 worktree 的补丁清单，而是基于 `docs/2026-04-18_全工作区代码审查报告.md` 中发现的共享根因，按“问题簇 -> 修复动作 -> 验收标准”组织的全仓工程修复路线。

本阶段已锁定的总体取向：

- 落地路径：`main-first`，以主工作区为唯一目标面，不以逐个修 worktree 再回并为主路径。
- 第一阶段目标：先稳共享骨架，包括实验合同、训练生命周期、DDP/provenance、正式后处理入口和实验线治理。
- 第一阶段验收：以工程验收为主，不把正式 full-dev 大规模复跑作为完成标准。
- 正式保留对象：只把 `event_probe_v2` 纳入第一阶段正式兼容路径。
- 代码组织：允许明显重切层次，新增共享 source-of-truth 模块，而不是只在原文件内局部补洞。
- 文档结构：按问题簇组织，并显式列出 deferred candidate 与 quarantined 研究线。

## 1. 目标公共接口

以下接口是后续实施时应当固定的目标形态。

### 1.1 实验合同

- 新增或升级统一的 `experiment_contract` 概念，作为训练、评测、API 评测、复现 suite、audit、postprocess 的共享实验合同。
- 现有 `compare` / `comparable_contract_hash` 不再作为唯一合同本体，而是从 `experiment_contract` 派生出的可比较子集。
- `experiment_contract` 至少覆盖：`stage_mode`、prompt/few-shot 配置、research split、semantic version、protocol hash、role alias hash、canonical metric mode、`postprocess_profile`、runtime/provenance 关键字段。

### 1.2 阶段模式命名

- `two_stage_typed` 改名为 `two_stage_per_type`。
- `two_stage_per_type` 的正式语义是：stage1 预测事件类型，stage2 按预测出的每个事件类型分别执行抽取，再合并结果。
- `two_stage_typed` 只作为迁移期旧名处理，不再作为新文档、新配置和新测试中的目标名字。

### 1.3 训练恢复命名

- 直接下线旧接口 `resume_from_checkpoint`。
- 新增 `resume_training_from`：表示恢复完整训练状态，必须包含 model、optimizer、scheduler、global step 等连续训练状态。
- 新增 `warm_start_from_checkpoint`：表示只加载模型/LoRA 权重作为新训练起点，不恢复优化器、调度器和 global step。
- 两者必须在 manifest 中分别记录，不允许再用模糊的 resume 字段覆盖两种语义。

### 1.4 后处理接口

- 新增正交字段 `postprocess_profile`。
- `pipeline_mode` 继续只表达结构化抽取流水线，例如 `e2e`、`cat_lite`、`record_corrector`。
- `postprocess_profile` 表达正式后处理配置档，第一阶段目标取值至少包含 `none` 与 `event_probe_v2`。
- `event_probe_v2` 应进入正式评测、summary、manifest、suite 和 `experiment_contract`，不再依赖旁路 rerank summary 作为正式结果来源。

## 2. 问题簇一：共享实验合同漂移

### 现状与风险

当前同一实验语义在多个入口重复定义：

- 训练侧、API 评测侧和 audit 侧对 `two_stage_typed` 的理解不一致。
- semantic version 已写入部分 summary/manifest，但没有进入 compare fail-fast。
- `meta.api_response_models`、few-shot 统计、canonical diagnostics、stage 统计等字段存在口径漂移。
- rerank sidecar 结果无法稳定参与正式 suite 和 compare contract。

结果是：即使多个实验产物看似有相同配置名，它们也未必可比较；即使 summary 字段存在，也未必代表真实运行语义。

### 修复动作

1. 建立 `experiment_contract` 作为唯一合同构造层。
2. 把 `compare` / `comparable_contract_hash` 改为从 `experiment_contract` 派生。
3. 将 prompt builder、parser、normalization、protocol、role alias、postprocess profile、stage mode 统一纳入 fail-fast 维度。
4. 将训练、`evaluate.py`、`evaluate_api.py`、repro suite、audit 脚本改为消费同一合同构造函数。
5. 将 `two_stage_per_type` 的训练展开、推理执行、summary/audit 统计统一到同一语义。

### 验收标准

- 同一配置在训练、本地评测、API 评测和 suite 中生成相同核心 `experiment_contract` 字段。
- prompt/parser/normalization version 变化会导致合同 hash 变化并触发 fail-fast。
- `two_stage_per_type` 的训练样本数、stage2 event-type 分解、few-shot 统计和评测执行路径一致。
- summary、run manifest、suite summary 中都能看到同一 `experiment_contract` 摘要。
- 现有 compare hash 兼容层仍可供旧脚本读取，但不再是唯一可信依据。

## 3. 问题簇二：训练生命周期语义错误

### 现状与风险

当前训练链路存在两类语义错误：

- `train_fit` 在入口和 trainer 内被重复应用，导致 `teacher_silver` 样本默认可能被第二次过滤掉。
- `resume_from_checkpoint` 名义上表示 resume，但实际只加载 LoRA 权重，没有恢复 optimizer、scheduler 和 global step。

这会直接改变模型实际看到的数据，以及后续实验对“继续训练”“阶段串联”“重启恢复”的理解。

### 修复动作

1. 将 train-fit 选择收敛到单一 owner，避免入口和 trainer 双重过滤。
2. 明确定义 teacher-silver 注入位置：先确定 gold train-fit，再按规则追加 teacher-silver，并记录 silver 来源、数量、hash 和过滤统计。
3. 下线 `resume_from_checkpoint`。
4. 实现 `resume_training_from` 与 `warm_start_from_checkpoint` 两个独立配置语义。
5. 在 manifest 中记录恢复类型、来源路径、来源 hash、是否恢复 optimizer/scheduler/global step。

### 验收标准

- 默认 frozen split 下，teacher-silver 不会因为 id 不在 `fit_ids` 中被静默清空。
- gold train-fit 数量、teacher-silver 数量、最终训练输入数量在日志和 manifest 中一致。
- `warm_start_from_checkpoint` 不恢复 global step，并在 manifest 中明确标记为新训练起点。
- `resume_training_from` 必须恢复训练状态；如果 checkpoint 缺少必要状态，应失败而不是退化为 warm-start。
- 正式配置、CLI、文档和测试中不再使用 `resume_from_checkpoint`。

## 4. 问题簇三：DDP / torchrun / provenance 不可信

### 现状与风险

当前 wrapper 和 trainer 对分布式运行的记录与真实执行不一致：

- wrapper manifest 在真正进入 `torchrun` 前写入，`world_size` 等字段可能永远是单进程视角。
- 静态 DPO cache 和 debug jsonl 会被多个 rank 共同删除、重建或覆盖。
- run manifest 同时承载 launcher、resolved config、runtime topology、artifact lineage，职责混乱。

这些问题会让多卡训练产物不可追溯，也会让缓存和 debug 文件在并发写入时不稳定。

### 修复动作

1. 将 provenance 分为三层：launcher manifest、runtime manifest、artifact/contract manifest。
2. launcher 层记录原始命令、原始 config、git/worktree 状态、启动环境。
3. runtime 层记录 resolved config hash、真实 distributed topology、effective stage/task、rank/world size。
4. artifact 层记录 checkpoint、snapshot、summary、protocol/alias hash、postprocess profile、`experiment_contract`。
5. 静态 cache、debug 样本日志和 snapshot 写入必须 rank-safe：只允许单一 owner 写共享产物，或按 rank 分目录再聚合。

### 验收标准

- 单卡与 `torchrun` 运行的 manifest 都能反映真实 world size 和 rank 信息。
- 多 rank 不会共享破坏性删除同一个 cache 目录。
- debug 产物不会被多个 rank 互相覆盖。
- resolved config hash 与训练实际使用配置一致。
- 训练 snapshot 与普通权重 artifact 在 manifest 中清楚区分。

## 5. 问题簇四：正式评测入口与 `event_probe_v2` 收口

### 现状与风险

`event_probe_v2` 是当前唯一明确需要保留的实验线，但它仍以 sidecar/rerank 形态存在：

- 正式评测入口不能直接产出 `event_probe_v2` 结果。
- sidecar summary 缺少完整 compare contract、run manifest 和 artifact lineage。
- summary metrics 与 diagnostics 可能使用不同 canonicalization 口径。

这会让最值得保留的后处理能力仍然停留在“可跑但不可审计”的状态。

### 修复动作

1. 新增 `postprocess_profile` 维度，并把 `event_probe_v2` 接入本地评测和 API 评测入口。
2. 将 `event_probe_v2` 的规则版本、profile 名、输入 artifact、输出 artifact 纳入 `experiment_contract`。
3. 将 canonicalization、metrics、diagnostics 的输入口径统一，避免同一 summary 内部描述两套预测。
4. 将旧 rerank sidecar 退为内部/研究入口；若保留，必须产出与正式入口一致的 manifest 和合同字段。
5. 将 `event_probe_v3` 从正式入口移除，不再作为正式 `postprocess_profile` 或公开 mode。

### 验收标准

- `evaluate.py` 和 `evaluate_api.py` 均可通过 `postprocess_profile=event_probe_v2` 产出正式 summary。
- summary、diagnostics、metrics、manifest、suite summary 均记录 `postprocess_profile`。
- `postprocess_profile=none` 与 `event_probe_v2` 的结果可被 suite 识别为不同合同，不能误汇总。
- `event_probe_v3` 无法通过正式 CLI/help/config 进入正式评测路径。
- 保留的内部 sidecar 不再被正式文档描述为主结果入口。

## 6. 问题簇五：实验分支治理失序

### 现状与风险

多个 worktree 已经有阶段性研究结论，但代码入口仍让它们看起来像正式能力：

- `event_probe_v3` 已被研究结论否决，但底层函数仍可直接调用。
- `candidate-table-v2-recall` 与 `record-assembly-v1` 指标退化或 correctness 风险明显。
- `candidate-table-v2` 和结构化 SFT 仍有研究价值，但未满足正式兼容入口的工程条件。

如果不显式治理，后续很容易把已否决逻辑重新混入正式结果。

### 修复动作

1. 将实验线分为三类：`promoted`、`deferred candidate`、`quarantined`。
2. `promoted`：第一阶段仅包含 `event_probe_v2`。
3. `deferred candidate`：包含 `candidate-table-v2` 与结构化 SFT 深化。
4. `quarantined`：包含 `event_probe_v3`、`candidate-table-v2-recall`、`record-assembly-v1`。
5. 从正式 CLI、正式配置、主 wrapper、正式文档中移除 quarantined 线；实现可保留为内部研究代码，但必须显式标注不可用于正式结果。

### 验收标准

- 正式入口不会暴露 quarantined mode。
- 默认配置和帮助文本不会把 quarantined 线描述为正式可比方案。
- deferred candidate 有明确重入 gate。
- 后续若要重新启用 deferred candidate，必须先满足 gate，而不是直接恢复 CLI 入口。

## 7. Deferred Candidate 重入 Gate

`candidate-table-v2` 和结构化 SFT 深化暂不进入第一阶段正式兼容路径，但可以在满足以下 gate 后重新讨论：

1. 有正式入口：不再依赖旁路脚本作为唯一产物来源。
2. 有完整 `experiment_contract`：包含 stage/task、postprocess、protocol、alias、semantic version、artifact lineage。
3. 有 summary 一致性：metrics、diagnostics、canonicalization 使用同一预测口径。
4. 有 manifest 一致性：训练、评测、postprocess 均能追溯输入 artifact 和 git/worktree 状态。
5. 有负向测试：覆盖错配、重复、schema 外角色、缺失 id、截断 preview、非 repo root 执行等高风险路径。
6. 有代表性验证：至少通过小规模 gate set，证明没有明显回归，再进入 full-dev 正式验证。

## 8. Quarantined 清单

以下路线第一阶段不允许作为正式入口或正式配置出现：

- `event_probe_v3`：可保留内部规则拆解，但不能作为正式 profile/mode 暴露；未来若做 `v3b`，必须先拆出安全数量规则并删除时间角色自动重判。
- `candidate-table-v2-recall`：文档级 recall 破坏 record 边界，暂不继续。
- `record-assembly-v1`：singleton broadcast 和 source text fallback 风险未解决，暂不继续。

## 9. 第一阶段执行顺序

建议按以下顺序实施，避免后续改动建立在不稳定合同上：

1. 建立 `experiment_contract` 与接口命名迁移。
2. 修正训练生命周期语义，包括 train-fit owner、teacher-silver、resume/warm-start。
3. 重切 provenance 与 DDP-safe artifact 写入。
4. 接入 `postprocess_profile` 和 `event_probe_v2`。
5. 移除 quarantined 线的正式入口。
6. 补齐 suite、audit、manifest、wrapper、summary 的合同一致性测试。
7. 更新工程文档，说明第一阶段只做工程验收，正式复跑另行安排。

## 10. 测试与验收矩阵

| 问题簇 | 必须覆盖的测试 |
|---|---|
| 实验合同 | `experiment_contract` 字段完整性、hash 稳定性、version 漂移 fail-fast、`two_stage_per_type` 训练/评测/audit 一致性 |
| 训练生命周期 | teacher-silver 保留、train-fit 单 owner、`resume_training_from`、`warm_start_from_checkpoint`、旧名下线 |
| DDP/provenance | 单卡与 `torchrun` manifest 差异、rank-safe cache、rank-safe debug artifact、resolved config hash |
| 后处理收口 | `postprocess_profile=none/event_probe_v2` 的 summary、diagnostics、contract、suite 汇总差异 |
| 实验治理 | quarantined mode 不出现在正式 CLI/help/config/docs，deferred candidate gate 有明确检查项 |

## 11. 非目标

- 第一阶段不承诺 full-dev 正式大规模复跑。
- 第一阶段不把 `candidate-table-v2` 或结构化 SFT 深化升格为正式能力。
- 第一阶段不继续扩展 `event_probe_v3`。
- 第一阶段不修复所有 worktree 的局部实现，只在 `main` 中吸收必要的共享修复和被保留能力。
