# OG-LANS 阶段性研究汇报：工作区、基线与当前结论

更新时间：2026-04-18  
适用范围：DuEE-Fin 中文金融文档级事件抽取实验  
说明：本报告同时汇总本地仓库与 4090 服务器上的正式实验结果。用户消息中出现的 `/data/TJK/og-lan/` 应视为笔误，实际使用路径为 `/data/TJK/og-lans/`。

## 摘要

本阶段的研究目标，是在统一评测体系下比较三类路线：一是本地 `Qwen3-4B` 小模型基线，二是外部 API 基线，三是围绕结构化训练、候选表构造、记录组装和事件特定后处理所开展的一系列 worktree 实验。当前最强的正式本地结果，是“主线训练 checkpoint + 保守事件特定后处理 v2”，其 `doc_role_micro_f1=0.5462`、`legacy_dueefin_overall_f1=0.5870`。若限定在当前统一 `dev` 口径下，当前最强的 API 正式结果为 `two-stage + few-shot + dynamic`，其 `doc_role_micro_f1=0.6586`、`legacy_dueefin_overall_f1=0.7081`。这说明：评测体系已经能够识别出高质量方案，但本地小模型体系与 API 基线之间仍存在明显差距。

从研究过程看，当前最有效的改进方向不是“继续增加复杂提示词”或“直接堆叠偏好优化”，而是更保守地沿着“结构化候选召回 + 记录级组装 + 事件特定、可审计的后处理修正”推进。其中，`event_probe_v2` 是目前唯一在 full-dev 正式口径下稳定抬升本地主指标的后处理版本；而 `event_probe_v3` 因时间角色重分配过于激进，已在正式验证中退化，不应继续沿原形态扩展。

## 一、指标说明

### 1.1 统一记号

对于任一评价单元，如果记：

- \(TP\)：预测正确的数量
- \(N_{\text{pred}}\)：预测总数
- \(N_{\text{gold}}\)：金标总数

则精确率、召回率与 F1 的统一形式为：

$$
P=\frac{TP}{N_{\text{pred}}},\qquad
R=\frac{TP}{N_{\text{gold}}},\qquad
F1=\frac{2PR}{P+R}.
$$

不同指标的区别，不在于公式本身，而在于“什么被当作一个评价单元”。

### 1.2 一个统一的计算示例

为了便于说明，下面使用一个小型示例。设评测集中共有 3 篇文档：

| 文档 | 金标事件 | 预测事件 |
|---|---|---|
| 文档 A | 1 条“企业融资”事件，包含 2 个正确论元 | 预测完全正确 |
| 文档 B | 2 条“股东减持”事件，每条各有 2 个论元，共 4 个论元 | 只正确预测其中 1 条事件 |
| 文档 C | 无金标事件 | 误预测出 1 条事件，且该事件论元不被原文支持 |

由此得到：

- 角色级总数：金标 6 个，预测 6 个，正确 4 个
- 事件实例总数：金标 3 条，预测 3 条，正确 2 条
- 事件类型总数：金标类型 2 个，预测类型 3 个，正确类型 2 个
- 单事件文档：仅文档 A
- 多事件文档：仅文档 B
- 发生幻觉的文档：仅文档 C

后文所有指标都基于这同一个例子演示。

### 1.3 文档级角色抽取 F1（`doc_role_micro_f1`）

中文释义：把每篇文档中的“事件类型 - 角色 - 论元值”三元组视为最小计数单元，在整个评测集上做微平均 F1。该指标是当前仓库默认主指标，也最能直接反映“模型是否把角色和值抽对了”。

计算公式：

$$
P_{\text{role}}=\frac{TP_{\text{role}}}{N_{\text{pred, role}}},\qquad
R_{\text{role}}=\frac{TP_{\text{role}}}{N_{\text{gold, role}}},\qquad
F1_{\text{role}}=\frac{2P_{\text{role}}R_{\text{role}}}{P_{\text{role}}+R_{\text{role}}}.
$$

示例计算：

$$
TP_{\text{role}}=4,\quad
N_{\text{pred, role}}=6,\quad
N_{\text{gold, role}}=6.
$$

因此：

$$
P_{\text{role}}=\frac{4}{6}=0.667,\quad
R_{\text{role}}=\frac{4}{6}=0.667,\quad
F1_{\text{role}}=0.667.
$$

### 1.4 文档级事件实例 F1（`doc_instance_micro_f1`）

中文释义：把一条完整事件记录视为一个评价单元。只有当事件类型及其整条记录中的角色和值都对应正确时，才算一个正确事件实例。该指标比角色级 F1 更严格，因为它要求一整条记录被完整拼对。

计算公式：

$$
P_{\text{inst}}=\frac{TP_{\text{inst}}}{N_{\text{pred, inst}}},\qquad
R_{\text{inst}}=\frac{TP_{\text{inst}}}{N_{\text{gold, inst}}},\qquad
F1_{\text{inst}}=\frac{2P_{\text{inst}}R_{\text{inst}}}{P_{\text{inst}}+R_{\text{inst}}}.
$$

示例中：

$$
TP_{\text{inst}}=2,\quad
N_{\text{pred, inst}}=3,\quad
N_{\text{gold, inst}}=3,
$$

所以：

$$
F1_{\text{inst}}=0.667.
$$

### 1.5 单事件文档 F1（`single_event_doc_role_micro_f1`）

中文释义：只在金标事件数等于 1 的文档子集上，计算角色级微平均 F1。它主要反映“模型在简单文档上是否稳定”。

计算公式仍与角色级 F1 相同，但只在单事件文档集合 \(\mathcal{D}_{\text{single}}\) 上统计：

$$
F1_{\text{single}}=
\frac{2P_{\text{single}}R_{\text{single}}}{P_{\text{single}}+R_{\text{single}}}.
$$

在示例中，只有文档 A 属于单事件文档，且预测完全正确，因此：

$$
F1_{\text{single}}=1.000.
$$

### 1.6 多事件文档 F1（`multi_event_doc_role_micro_f1`）

中文释义：只在金标事件数大于等于 2 的文档子集上，计算角色级微平均 F1。它主要反映“模型在篇章内多事件并存时的区分与组装能力”。

示例中，只有文档 B 属于多事件文档，其角色级统计为：

$$
TP_{\text{multi}}=2,\quad
N_{\text{pred, multi}}=2,\quad
N_{\text{gold, multi}}=4.
$$

故：

$$
P_{\text{multi}}=1.000,\quad
R_{\text{multi}}=0.500,\quad
F1_{\text{multi}}=\frac{2\times1.000\times0.500}{1.000+0.500}=0.667.
$$

### 1.7 DuEE-Fin 旧式 overall F1（`legacy_dueefin_overall_f1`）

中文释义：这是为了与较早 DuEE-Fin 论文表格保持可比而保留的一条旧式轨道。它对应严格事件级 F1，可以理解为“事件类型和全部论元都必须完全匹配”的总体 F1。

其公式仍是：

$$
P_{\text{legacy}}=\frac{TP_{\text{strict}}}{N_{\text{pred, strict}}},\qquad
R_{\text{legacy}}=\frac{TP_{\text{strict}}}{N_{\text{gold, strict}}},\qquad
F1_{\text{legacy}}=\frac{2P_{\text{legacy}}R_{\text{legacy}}}{P_{\text{legacy}}+R_{\text{legacy}}}.
$$

在本报告的统一示例里，它与文档级事件实例 F1 恰好相同：

$$
F1_{\text{legacy}}=0.667.
$$

但在真实实验中，这一指标与 `doc_instance_micro_f1` 的组织方式、输出结构和论文表格用途并不完全等价，因此报告中仍单独列出。

### 1.8 事件类型识别 F1（`type_f1`）

中文释义：只考察模型是否识别出正确的事件类型，不考察具体角色和值。它更接近“先判断文档里出现了哪些事件类型”的能力。

公式为：

$$
P_{\text{type}}=\frac{TP_{\text{type}}}{N_{\text{pred, type}}},\qquad
R_{\text{type}}=\frac{TP_{\text{type}}}{N_{\text{gold, type}}},\qquad
F1_{\text{type}}=\frac{2P_{\text{type}}R_{\text{type}}}{P_{\text{type}}+R_{\text{type}}}.
$$

示例中：

$$
TP_{\text{type}}=2,\quad
N_{\text{pred, type}}=3,\quad
N_{\text{gold, type}}=2.
$$

因此：

$$
P_{\text{type}}=\frac{2}{3},\quad
R_{\text{type}}=1,\quad
F1_{\text{type}}=0.800.
$$

### 1.9 幻觉样本率（`hallucination_rate`）

中文释义：在全部样本中，预测结果包含“原文不支持的论元值”或“明显脱离原文的事件内容”的文档比例。它不是 F1，而是一个风险诊断指标。

计算公式：

$$
\text{HallucinationRate}=\frac{N_{\text{hallucinated samples}}}{N_{\text{all samples}}}.
$$

示例中，只有文档 C 出现了幻觉，因此：

$$
\text{HallucinationRate}=\frac{1}{3}=0.333.
$$

## 二、研究主线与工作区全景

### 2.1 总体研究逻辑

本项目的核心问题不是“模型能否输出 JSON”，而是“在长文本、多事件、多角色值干扰同时存在时，模型是否还能稳定输出与金融事件 schema 一致、并且可以被严格评测的记录”。围绕这个问题，当前研究大致形成了四条线：

1. **基线线**：先明确本地 `Qwen3-4B` 与 API 模型在统一评测下能达到什么水平。
2. **训练主线**：把事件抽取任务分解为角色、跨度、候选表等更结构化的训练任务，再逐步回到完整事件记录。
3. **候选与组装线**：尝试先召回候选，再组装记录，而不是一次性自由生成整条事件。
4. **事件特定后处理线**：在不破坏统一评测协议的前提下，对高频错误类型做保守、可审计的规则修正。

### 2.2 当前全部 worktree 的研究定位

## 四、各方案的阶段分析

### 4.1 本地 `Qwen3-4B` 基线

本地小模型基线承担的任务，是回答两个问题：第一，完全不训练时，小模型能达到什么水平；第二，单阶段和两阶段、零样本和少样本，哪种组合更适合小模型。

当前正式结果表明：

- 单阶段零样本是本地基线中最强的组合，`doc_role_micro_f1=0.3240`
- 两阶段零样本没有带来提升，反而略有下降到 `0.3071`
- 少样本对本地小模型并不稳定，尤其单阶段 few-shot 下降到 `0.0895`
- 幻觉样本率在零样本条件下很高，说明小模型即使能“猜中类型”，也很容易在角色和值上脱离原文

这一阶段的结论是：**本地小模型本身并不足以支撑高质量文档级事件抽取，必须依赖结构化训练或外部后处理约束。**

### 4.2 API 基线

API 基线的意义不是替代本地模型，而是提供“在同一评测协议下，这个任务大约能做到多好”的参考上限。当前正式 API 结果为：

- `doc_role_micro_f1=0.6586`
- `doc_instance_micro_f1=0.2337`
- `legacy_dueefin_overall_f1=0.7081`
- `single_event_doc_role_micro_f1=0.6612`
- `multi_event_doc_role_micro_f1=0.6777`
- `type_f1=0.8963`
- `hallucination_rate=0.0803`

这说明当前评测体系并没有系统性压低分数；相反，它能够在同一协议下识别出明显更强的方案。因此，本地体系与 API 之间的差距，更可能来自模型能力、训练策略和结构化建模方式，而不是来自评测不合理。

在这一正式基线之上，本阶段又补做了一轮更激进的 API 改进探索，希望进一步提升指标。核心思路包括两类：

- **按事件类型拆分第二阶段抽取**：先预测事件类型，再按类型分别调用抽取，而不是把缩小后的 schema 一次性喂给模型
- **对比式 few-shot 检索**：在动态 few-shot 的基础上，额外加入“易混淆提醒”，显式告诉模型不要把某些高风险角色混淆

这轮探索先在 `dev-20` smoke 上做了三组可比实验：

| 方案 | `doc_role_micro_f1` | `doc_instance_micro_f1` | `legacy_dueefin_overall_f1` | `single_event_doc_role_micro_f1` | `multi_event_doc_role_micro_f1` | `hallucination_rate` | 结论 |
|---|---:|---:|---:|---:|---:|---:|---|
| 当前 champion：`two-stage + dynamic` | 0.5785 | 0.2188 | 0.7047 | 0.5038 | 0.6981 | 0.0500 | 作为 smoke 参照系 |
| `two-stage + contrastive` | 0.5534 | 0.2188 | 0.6939 | 0.4648 | 0.6981 | 0.1000 | 退化 |
| `two-stage-typed + dynamic` | 0.5504 | 0.2353 | 0.6900 | 0.4658 | 0.6916 | 0.0500 | 退化 |
| `two-stage-typed + contrastive` | 0.5525 | 0.2388 | 0.6866 | 0.4690 | 0.6916 | 0.1000 | 退化 |

这组实验给出一个很明确的阶段结论：**这轮 API 提升方案没有通过 gate，不应继续放大到 `dev-100` 或 full `dev`。**

其中，退化主因并不是“对比式提醒”本身，而是“按事件类型拆分第二阶段抽取”这一层。其 `stage1_gold_coverage_rate=0.65`、`stage1_miss_rate=0.15`，说明第一阶段一旦漏掉事件类型，后续按类型拆分抽取就会被系统性带偏。因此，这条路线当前更适合视为“已完成验证但阶段性否决”的探索，而不是下一轮主推进方向。

### 4.3 `structured-sft-pipeline`：统一协议与训练基础设施

这个工作区没有单独产出一条可直接对比的正式主表分数，但它完成了若干非常关键的基础工作：

- 统一本地模型、API 模型和训练 checkpoint 的可比评测协议
- 引入 teacher-silver 数据增强管线
- 引入旧式 `legacy_dueefin` 轨道和单事件/多事件拆分口径
- 提供共享的记录修正与可审计诊断能力

从研究角度看，这个工作区的价值不在于“立刻涨分”，而在于为后续所有对比实验建立统一口径。没有这一层，后续 worktree 的数值很难被放到同一张表上讨论。

### 4.4 `sft-distill-role-span-table`：当前主训练线

这个工作区是当前本地主训练线的核心。它把训练过程拆解为更结构化的阶段，例如角色级、跨度级和候选表级任务，并最终形成了 `singlecard_table_full_s3407_e2` 这一正式 checkpoint。

当前最重要的正式结果是：

- 主线 checkpoint：`doc_role_micro_f1=0.5158`
- 单事件文档：`0.5964`
- 多事件文档：`0.4538`
- `legacy_dueefin_overall_f1=0.5592`

这个结果相对本地 `Qwen3-4B` 基线已经有明显提升，说明“结构化训练”是有效方向；但它与 API 基线之间仍存在大约 `0.19` 的主指标差距，说明仅靠当前训练主线还不足以逼近目标水平。

该工作区内部还做过两个关键派生实验：

- `candidate_table_from_table`：`0.5003`
- `candidate_table_from_span`：`0.4054`

这两个结果表明：**并不是所有“更结构化”的训练分解都会带来收益**。尤其从跨度直接到候选表的链路，当前还不稳定。

### 4.5 `candidate-table-v2`

这个工作区代表的是“先候选召回，再做候选表级修复和重排”的方向。它不是直接让模型输出整条事件记录，而是先把可能的候选值收集起来，再做更保守的修复与筛选。

在 gate200 上，这条线给出了目前最强的阶段性信号：

- `doc_role_micro_f1=0.5665`
- `multi_event_doc_role_micro_f1=0.4987`
- `doc_instance_micro_f1=0.1720`

这说明“候选表 + 保守修复”的确可能比当前主线 checkpoint 更适合多事件场景。但由于它尚未跑到 full-dev，因此当前还不能把它写成“已经超过本地主线”。现阶段更准确的表述是：**这是最值得继续完成 full-dev 验证的一条候选路线。**

### 4.6 `candidate-table-v2-recall`

这个工作区试图在 `candidate-table-v2` 的基础上进一步做保守召回增强，希望通过补回更多候选值来改善 recall。

结果显示：

- `doc_role_micro_f1=0.5304`
- `legacy_dueefin_overall_f1=0.5890`

相较于 `candidate-table-v2` 本身，这条线全面退化。这说明当前问题不只是“少召回了一些候选”，而是“错误候选进入后，后续修复和组装本身会被扰乱”。因此这条方向目前不建议继续原样推进。

### 4.7 `record-assembly-v1`

这个工作区想解决的问题是：既然候选已经有了，能否显式地做“记录组装”，而不是继续依赖较弱的重排逻辑。

其 gate200 结果为：

- `doc_role_micro_f1=0.5134`
- `multi_event_doc_role_micro_f1=0.4142`

这比 `candidate-table-v2` 和 `event_probe_v1/v2` 都要差。一个更直观的解释是：**记录组装本身比候选修补更难，一旦约束不足，就容易把候选拼错成错误记录。**

### 4.8 `event-probe-v1`

这个工作区是当前“事件特定后处理”路线的起点。它的思想不是大规模重写模型输出，而是只对少数高频错误事件类型做保守修补，例如：

- 质押 / 解除质押的类型配对
- 上市事件阶段标签归一
- 一些数值碎片、融资轮次等显著格式问题

它在 full-dev 上把主线 checkpoint 从 `0.5158` 提升到 `0.5348`。这说明：

1. 当前模型的确存在一些集中、可规则化的高频错误；
2. 只要规则足够保守，后处理不会明显破坏评测公平性，反而能提升真实记录质量。

### 4.9 `event-probe-v2-error-driven`

这个工作区是在 `event_probe_v1` 基础上，围绕高频错误角色进一步做“误差驱动”的小范围修补。它主要增强了：

- `股份回购` 中价格表达的规范化
- `亏损` 中金额碎片合并和谣言上下文抑制
- `中标` 中项目名称碎片的回拼

其正式 full-dev 结果为：

- `doc_role_micro_f1=0.5462`
- `single_event_doc_role_micro_f1=0.6167`
- `multi_event_doc_role_micro_f1=0.4965`
- `doc_instance_micro_f1=0.1260`
- `legacy_dueefin_overall_f1=0.5870`

这就是当前本地体系的正式最优结果。

同一工作区中还继续试做了一个 `v3` 版本，希望进一步修正质押类时间角色和部分数量角色。但 `v3` 的 full-dev 结果下降到 `0.5398`。问题集中在：

- `质押/事件时间`
- `质押/披露时间`
- `解除质押/事件时间`
- `解除质押/披露时间`

换言之，**这条线的失败不是因为“后处理思路不行”，而是因为时间角色自动重分配过于激进。**

### 4.10 `server-storage-migration`

这个工作区不追求实验指标，它解决的是服务器上的数据、模型和日志目录治理问题。其主要作用包括：

- 统一服务器上的数据目录到 `/data/TJK/og-lans/data`
- 统一模型目录到 `/data/TJK/og-lans/models`
- 统一训练与评测日志目录到 `/data/TJK/og-lans/logs`
- 保证 worktree、主仓和脚本都能在统一路径约束下复用

它的重要性在于：后续所有长训练、全量评测和基线复现，都依赖这个工程层的稳定性。

## 五、阶段性分析与判断

### 5.1 当前评测体系是否足够完备

就当前实验而言，评测体系已经具备较强的完备性，原因有三：

1. 它同时报告角色级、实例级、单事件/多事件、旧式 comparable track、类型识别和幻觉率，而不是只盯一个 F1。
2. 它能够在同一协议下区分出 API 基线、本地基线、主线训练和后处理版本的明显强弱。
3. 它已经支持统一的可比分析，不需要再依赖“换一个脚本可能更高”这类不稳定口径。

因此，当前的主要问题不在“怎么换指标”，而在“如何让本地模型真正接近更强方案”。

### 5.2 为什么 API 基线显著优于本地体系

从当前统一 `dev` 口径的正式结果看，API 基线相对本地正式最优结果有以下差距：

- `doc_role_micro_f1`：`0.6586` vs `0.5462`
- `legacy_dueefin_overall_f1`：`0.7081` vs `0.5870`
- `doc_instance_micro_f1`：`0.2337` vs `0.1260`

这说明差距不仅体现在“角色有没有抽对”，更体现在“能否把完整记录拼对”。简言之：

- 本地体系已经学会了一部分事件类型和角色值识别；
- 但在多事件分离、同类事件多记录区分、记录完整性上仍然较弱。

### 5.3 为什么 `event_probe_v2` 值得保留，而 `event_probe_v3` 不值得继续原形态推进

`event_probe_v2` 的特征是：只修补少数已经被误差分析明确指出、且能够被原文直接校验的错误模式。因此它提升了主指标，同时没有引入新的幻觉风险。

`event_probe_v3` 的问题在于：它试图进一步自动改判质押类的时间角色。虽然这种修补看起来更“聪明”，但一旦句级线索不稳定，就会把本来正确的时间角色从一个桶移动到另一个桶，导致大面积精度下降。

因此，本阶段的合理结论是：

- 保留 `event_probe_v2`
- 不合并 `event_probe_v3`
- 若继续做 `v3b`，也应只保留数量类的安全规则，而删除时间角色自动重判

### 5.4 当前最值得做的下一步是什么

如果以“投入产出比”而不是“概念复杂度”来判断，当前最值得继续推进的是：

1. **先把 `candidate-table-v2` 做 full-dev 正式验证**
2. **把 `event_probe_v2` 作为当前正式后处理版本固定下来**
3. **在 API 路线上保留 `two-stage + dynamic` 作为 champion，仅尝试更保守的高风险角色守门或选择性 verifier，而不继续推进 `two-stage-typed` 这一方向**
4. **继续沿‘候选召回 + 记录组装 + 保守后处理’三段式路线推进**

不建议优先做的方向包括：

- 继续扩展 `event_probe_v3` 这类激进时间规则
- 继续在 `candidate-table-v2-recall` 上做原样召回增强
- 继续放大本轮 `two-stage-typed / contrastive` API 路线
- 回到“只改 prompt 或只加更多 few-shot”的方向

## 六、当前结论

截至本报告撰写时，可以给出如下阶段性结论：

1. **API 基线已经达到较强水平**，说明当前统一评测体系能够支持高质量结果，不存在明显的“评测压分”问题。
2. **本地 `Qwen3-4B` 原始基线偏弱**，单靠提示设计难以接近可用水平。
3. **结构化训练主线有效**，它把本地正式结果从基线的 `0.3240` 提升到了 `0.5158`。
4. **事件特定、保守、可审计的后处理是当前最有效的本地增益来源**，其中 `event_probe_v2` 已把本地正式最优结果提升到 `0.5462`。
5. **候选表方案在 gate 阶段最有潜力**，`candidate-table-v2` 值得继续做 full-dev 验证。
6. **本轮 API 新方案 `two-stage-typed / contrastive` 已完成 smoke 验证，但未通过 gate**，不应继续放大到 `dev-100` 或 full `dev`。
7. **当前与文献对齐目标 `legacy_dueefin_overall_f1≈0.75` 仍有明显距离**，但 API 基线已经证明这一目标并非不可达。

## 附录一：术语解释

- **文档级事件抽取**：不是只在一句话里找事件，而是要在整篇文档中识别多个事件，并把角色和值组织成结构化记录。
- **单事件文档**：一篇文档中只有 1 条金标事件。
- **多事件文档**：一篇文档中有 2 条及以上金标事件。它比单事件文档更难，因为模型需要区分多条记录之间的边界。
- **角色级 F1**：把“事件类型 - 角色 - 论元值”看作最小单位来算 F1，更关注模型有没有把角色和值抽对。
- **事件实例 F1**：把一整条事件记录看作最小单位来算 F1，更关注模型有没有把完整记录拼对。
- **旧式 overall F1**：为了与早期 DuEE-Fin 论文表格保持可比而保留的严格事件级 F1。
- **事件类型识别 F1**：只看模型是否识别出了正确的事件类型，不看角色和值是否完整。
- **幻觉样本率**：预测结果中出现原文不支持内容的文档占比。
- **零样本（zeroshot）**：不给示例，只给任务说明，让模型直接抽取。
- **少样本（few-shot）**：在任务说明前后给少量示例，让模型模仿示例格式和抽取方式。
- **单阶段抽取**：模型一次性直接输出最终事件记录。
- **两阶段抽取**：先判断有哪些事件类型，再在限制的类型集合内抽取完整记录。
- **teacher-silver**：把高置信预测结果转成“银标数据”，再作为额外训练监督使用；它不等于人工金标。
- **候选表**：先列出每种事件、每个角色可能对应的候选值，再从中挑选和组装最终记录。
- **记录组装**：在候选值已经给出的情况下，进一步决定哪些值应被拼成同一条事件记录。
- **事件特定后处理**：针对某些高频错误事件类型做保守修补，例如数值碎片合并、角色标签归一、显著错配删除等。
- **SCV**：一种语义一致性检查机制，用来减少把“实际上仍被原文支持的样本”误当作错误负样本的风险。
- **LANS**：一种基于当前训练损失动态调节负样本难度的机制，目的是让训练过程中的负样本从易到难地变化。
- **CAT-lite**：一种较轻量的后处理/过滤机制，用于在评测时清理明显不合理的结构化输出。
- **schema compliance**：模型输出是否符合预定义事件类型与角色集合的约束。
- **gate200**：从训练集调参子集抽出的 200 条样本，用于低成本筛选方案；它不能替代正式 full-dev。
- **full-dev**：在正式开发集上进行的完整评测，是当前报告最重要的正式结果来源。
